| updated_at     | TIMESTAMP    |    |             | N    |        | CURRENT_TIMESTAMP | Updated Time |

### Relationships

supplier (1)
↓
supplier_contact (N)

### Business Rules

1. A supplier may have multiple contacts.
2. At least one primary contact should exist for active suppliers.
3. Contact information should be retained for historical transaction traceability.
4. Inactive contacts shall not be selectable in operational forms.

## 10. 编码规则与流水号生成

### 10.1 设计原则

业务编号由后台服务生成，数据库负责保存当前流水状态并通过唯一约束防止重复。

### 10.2 编号格式

| Object             | Prefix | Format                            | Example         |
| ------------------ | ------ | --------------------------------- | --------------- |
| Lot                | LOT    | LOT + YYYYMMDD + 4-digit sequence | LOT202606050001 |
| Quality Inspection | QI     | QI + YYYYMMDD + 4-digit sequence  | QI202606050001  |
| Purchase Order     | PO     | PO + YYYYMMDD + 4-digit sequence  | PO202606050001  |

### 10.3 Sequence Generator Table

`system.sequence_generator` stores current sequence values.

### 10.4 Generation Logic

1. Backend calls `generate_code(sequence_code)`.
2. Database locks the sequence row.
3. `current_value` increases by 1.
4. Backend returns formatted business code.
5. Target business table stores the generated code with a UNIQUE constraint.

### 10.5 Concurrency Control

Generation must use row-level locking or transaction control.
The system must not generate business codes by querying the latest existing record.
